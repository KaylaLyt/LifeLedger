## 上下文

该变更定义一个离线优先的 Android 个人生活账本应用，覆盖资产快照、送礼台账、轻量 Todo、本地密码锁和 JSON 导出。与常见的流水记账工具不同，本产品的核心行为是按任意周期录入一次“全量状态”，因此设计必须优先支持历史补录、按快照日期排序计算、单设备本地使用和长期可迁移的数据结构。

该应用只面向单个本地用户实例，不提供云同步、多账本协作或联网依赖。用户接受应用内部使用 SQLite/Room 保存数据，但要求能随时导出为单个 JSON 文件，并在应用停止维护后仍可阅读与迁移。

## 目标 / 非目标

**目标：**
- 提供多账户资产快照能力，支持任意日期补录、总资产变化查看、目标设定与达成判断。
- 提供送礼台账与生日提醒能力，并在生日前两周自动生成“准备礼物”Todo。
- 提供轻量 Todo，支持单次、常见循环和通过表单生成的高级循环规则。
- 提供应用级密码锁、失败预警和连续 5 次失败后的本地数据销毁。
- 提供单文件 JSON 导出，覆盖账户、快照、人员、礼物记录和 Todo 数据。

**非目标：**
- 不实现云同步、联网账号、跨设备自动备份或共享账本。
- 不实现高频逐笔流水记账、复杂统计报表、项目式任务管理或多人协作。
- 不提供手工编辑 JSON 后再导回的工作流。
- 不将负债自动并入总资产，也不对送礼台账做按人累计结算。

## 决策

### 1. 使用 SQLite/Room 作为主存储，JSON 作为导出格式

- 选择原因：Room 更适合处理跨模块关联、日期排序、筛选查询和数据一致性；JSON 导出负责可读性与长期迁移。
- 备选方案：直接以 JSON 文件作为主存储。
- 不选原因：直接写 JSON 会增加增量更新和查询复杂度，也更容易因写入失败导致整包损坏。

### 2. 资产记录采用“快照模型”而非“流水模型”

- 选择原因：用户按数月记录一次完整余额与周期汇总，快照模型更符合真实习惯，界面和统计也更简单。
- 备选方案：以流水账为中心，再按周期汇总。
- 不选原因：会引入大量不必要的录入负担和分类复杂度，偏离核心使用场景。

### 3. 总资产变化以快照日期排序计算，允许补录历史

- 选择原因：用户计划补录最近三年数据，系统必须以 `snapshotDate` 重建真实时间线，而不是依赖录入时间。
- 备选方案：仅按创建时间计算趋势。
- 不选原因：一旦补录旧数据，变化趋势和目标判断就会失真。

### 4. 负债作为独立记录域，不自动参与总资产汇总

- 选择原因：用户短期内只需要“有地方记”，不希望负债影响主视图总资产。
- 备选方案：将负债以负数自动并入总资产。
- 不选原因：会让主视图与用户现有认知不一致，且削弱资产快照的直观性。

### 5. Todo 循环规则提供“常见周期 + 高级表单”两层入口

- 选择原因：普通用户入口应足够轻量，但仍需要支持接近 cron 的灵活表达。
- 备选方案：仅支持固定周期，或直接暴露 cron 文本输入框。
- 不选原因：仅固定周期不够灵活；直接输入 cron 虽强大，但移动端可用性差且容易出错。

### 6. 本地提醒由规则计算下一次触发时间，不承诺服务器级精确调度

- 选择原因：Android 后台调度受系统省电与权限限制，应优先保障实现可行性与行为稳定性。
- 备选方案：追求任意复杂规则的绝对精确触发。
- 不选原因：会高估系统能力，且增加实现和维护成本。

### 7. 安全边界采用“界面锁 + 失败 5 次销毁本地数据”

- 选择原因：用户需要基础防护和极端情况下的自毁能力，但不要求复杂加密体系。
- 备选方案：仅锁定不销毁，或引入数据库级加密。
- 不选原因：仅锁定不满足用户要求；数据库级加密超出当前范围与复杂度预算。

### 8. 敏感信息默认受控展示，完整号码仅在详情中暴露与复制

- 选择原因：完整卡号必须保留，但不应在普通列表中直接暴露。
- 备选方案：所有页面都直接显示完整号码，或只保存后四位。
- 不选原因：前者风险过高；后者不满足用户需求。

## 风险 / 权衡

- [密码失败 5 次即清空本地数据，误操作风险较高] -> 每次失败都显示剩余次数，并在最后一次前给出明确不可恢复警告。
- [高级循环规则比普通 Todo 更复杂] -> 通过表单约束可配置范围，并在保存前展示下一次触发时间预览。
- [补录历史快照会影响既有趋势] -> 所有统计统一基于快照日期重算，并在补录后刷新变化结果。
- [完整卡号属于敏感信息] -> 列表默认遮挡，仅详情页显示与复制，并由应用锁保护入口。
- [JSON 导出结构需长期兼容] -> 导出根对象包含 `schemaVersion` 和 `exportedAt`，后续升级以版本字段演进。

## Migration Plan

1. 初始化 Room 数据库与基础实体，定义账户、快照、花销汇总、人员、礼物记录、Todo、提醒规则和应用锁设置。
2. 先交付核心录入链路：账户管理、资产快照、礼物记录、Todo 列表。
3. 在核心数据稳定后接入本地提醒、生日派生 Todo 和高级循环规则生成。
4. 最后补充密码失败计数、销毁流程和单文件 JSON 导出。
5. 当前无存量线上数据需要迁移；若后续修改导出结构，使用 `schemaVersion` 驱动兼容逻辑。
