## 新增需求

### 需求:联系人生日必须可选

系统必须允许联系人在不填写生日的情况下被创建与保存；`PersonEntity.birthdayMonth` 与 `birthdayDay` 必须均支持 `null` 值用于表达"未填写"。

#### 场景:新增联系人时留空生日
- **当** 用户在新增联系人对话框中留空"生日月"与"生日日"输入框并点击保存
- **那么** 系统必须成功保存该联系人，且该记录的 `birthdayMonth` 与 `birthdayDay` 必须为 `null`

#### 场景:编辑联系人时清空生日
- **当** 用户在编辑联系人对话框中将原本有值的生日月 / 日输入框清空并点击保存
- **那么** 系统必须将对应记录的 `birthdayMonth` 与 `birthdayDay` 更新为 `null`

#### 场景:联系人卡片隐藏空生日行
- **当** 系统渲染 `birthdayMonth` 为 `null` 的联系人卡片
- **那么** 卡片必须不显示"生日 X月X日"这一行；其它字段（姓名、关系、备注、净人情金额）保持正常显示

#### 场景:生日临近染色对无生日联系人无效
- **当** 某个联系人的 `birthdayMonth` 或 `birthdayDay` 为 `null`
- **那么** `isBirthdayApproaching` 必须返回 `false`，该联系人姓名颜色必须使用默认 `onSurface` 色，禁止走"生日临近"的高亮染色

### 需求:账户必须支持编辑

系统必须允许用户在不新建账户的情况下修改已有账户的名称、类型、完整号码、备注、是否计入总资产五个字段；编辑后的账户名变化必须自动反映在所有历史快照的展开视图中（通过 id 关联）。

#### 场景:通过详情弹窗进入编辑
- **当** 用户点击资产 Tab 中的某个账户卡片打开详情弹窗
- **那么** 详情弹窗必须显示 `[编辑]` 按钮；点击该按钮必须打开编辑账户对话框，表单预填当前账户的所有字段

#### 场景:保存后字段立即更新
- **当** 用户在编辑账户对话框中修改字段并点击保存
- **那么** 系统必须更新对应 `AccountEntity` 记录，资产 Tab 的账户列表、历史快照展开视图中的账户名必须立即反映变更

### 需求:账户必须支持软删（归档）

系统必须允许用户将账户标记为"已归档"而非物理删除；`AccountEntity` 必须新增 `archived: Boolean` 字段，默认 `false`。归档操作必须在资产 Tab 的账户详情弹窗中通过 `[归档]` 按钮触发，且必须进行一次轻量二次确认。

#### 场景:归档需要轻量二次确认
- **当** 用户在账户详情弹窗中点击 `[归档]` 按钮
- **那么** 系统必须弹出二次确认对话框，文案提示"账户将不再出现在列表中，历史快照将保留"；用户确认后系统必须将该账户的 `archived` 字段置为 `true`

#### 场景:归档账户从活跃列表中消失
- **当** 某账户的 `archived` 为 `true`
- **那么** 资产 Tab 的账户列表、新增快照时的余额输入表、资产趋势折线图的账户线必须不包含该账户

#### 场景:归档账户在历史快照中仍可见
- **当** 用户展开包含已归档账户余额的历史快照
- **那么** 该账户的名称与余额必须仍然展示，且账户名后必须附带 `(已归档)` 标识以提示用户

#### 场景:系统禁止账户硬删
- **当** 用户操作 UI 寻找"删除账户"的入口
- **那么** 系统必须不提供任何"硬删账户"（从数据库物理删除 AccountEntity 记录）的交互入口；唯一的移除方式是归档

### 需求:新增同名归档账户必须询问恢复

系统必须在用户新增账户时检测名称是否与任何已归档账户冲突；若命中，系统必须弹出对话框询问是否恢复该归档账户，而不是静默创建或静默恢复。

#### 场景:命中归档账户时弹出恢复询问
- **当** 用户新增账户，名称与某条 `archived=true` 的历史账户完全相同（不区分大小写与前后空白）
- **那么** 系统必须弹出对话框，内容包含"已存在同名归档账户，是否恢复？"；用户选择恢复后，系统必须将该归档账户的 `archived` 置为 `false`，并用新输入覆盖其它字段

#### 场景:用户取消恢复
- **当** 用户在恢复对话框中点击取消
- **那么** 系统必须不创建新账户，也必须不修改任何已归档账户；用户的输入保留在新增账户对话框中以便重新调整

#### 场景:名称与活跃账户冲突
- **当** 用户新增账户，名称与某条 `archived=false` 的活跃账户完全相同
- **那么** 系统必须阻止保存并提示"该名称已被使用"

### 需求:资产快照必须支持真删

系统必须允许用户从历史快照列表中删除任意快照记录；删除必须为物理删除，且必须通过 FK `onDelete = CASCADE` 连带删除该快照关联的所有 `SnapshotBalanceEntity` 与 `SnapshotExpenseEntity`。删除操作必须经过二次确认。

#### 场景:通过快照卡片触发删除
- **当** 用户点击某条历史快照卡片右上角的删除图标
- **那么** 系统必须弹出二次确认对话框，文案提示"确定要删除这条快照吗？关联的余额和花销记录也会被删除。"

#### 场景:确认删除后级联清理
- **当** 用户在二次确认对话框中点击确认
- **那么** 系统必须从数据库中物理删除对应的 `SnapshotEntity`，且该快照关联的所有 `SnapshotBalanceEntity` 与 `SnapshotExpenseEntity` 必须被同步删除；UI 上的快照列表与资产趋势图必须立即反映变更
