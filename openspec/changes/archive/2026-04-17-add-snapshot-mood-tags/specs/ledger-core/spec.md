## 新增需求

### 需求:快照情绪字段
系统必须允许在资产快照上记录一个 1..5 的整数情绪值（`mood`），以及"未填写"状态（`null`）。UI 层必须用 Emoji 五档（😢=1 / 😕=2 / 😐=3 / 🙂=4 / 🤩=5）渲染，未填写时显示占位或留空；底层存储必须保持为整数以支持未来的数值化分析。

#### 场景:新建快照时选择情绪
- **当** 用户在「新增资产快照」对话框点击情绪选择器并选中 🙂（档位 4）
- **那么** 保存后的快照 `mood` 字段等于 `4`
- **并且** 快照卡片上必须显示 🙂 图标

#### 场景:不填写情绪
- **当** 用户新建快照时未与情绪选择器交互
- **那么** 保存后的快照 `mood` 字段必须为 `null`
- **并且** 快照卡片上必须不显示情绪图标（显示空位或省略整行）

#### 场景:非法值被拒绝
- **当** 任何写入路径（UI、导入）传入不在 `1..5` 且不为 `null` 的情绪值
- **那么** 系统必须将其归一化为 `null`，并记录一次导入告警（如果来自导入）

---

### 需求:快照批注字段仅限情绪 / 标签 / 心得可编辑
快照的金额类字段（账户余额、花销、目标、负债、日期）在保存后禁止修改，仅删除操作有效。但是情绪（`mood`）、关联标签（`tags`）、心得（`note`）三项属于"批注"类字段，系统必须允许用户在任意时刻打开「编辑批注」入口对它们单独修改。编辑批注时禁止改动任何金额类字段。

#### 场景:打开编辑批注
- **当** 用户在快照卡片上点击「编辑批注」图标
- **那么** 系统必须弹出批注编辑对话框，仅显示情绪选择器、标签输入、心得多行文本三项
- **并且** 对话框中必须不提供编辑余额、目标、负债、花销的入口

#### 场景:保存批注修改
- **当** 用户在批注编辑对话框中修改情绪、标签、心得并点击「保存」
- **那么** 系统必须通过 `updateSnapshotAnnotation(snapshotId, mood, note, tagIds)` 专用路径更新
- **并且** 快照的 `snapshotDate` / `createdAt` / 余额 / 花销 / 目标 / 负债等金额类字段必须保持不变

#### 场景:禁止通过批注路径篡改金额
- **当** 任何非「删除」路径试图修改快照余额或花销记录
- **那么** 系统必须拒绝并保持原数据不变

---

### 需求:标签独立表与软删
系统必须用独立表 `tags` 保存标签（字段至少包含 `id`、`name`、`archived`、`sortOrder`），名称必须在大小写不敏感下唯一。当标签被任意历史快照引用时，系统必须禁止硬删，只能归档（`archived=1`）。已归档标签不得出现在新建/编辑快照时的候选下拉里，但必须继续出现在被引用的历史快照上。

#### 场景:创建新标签
- **当** 用户在标签输入框中输入一个系统中尚不存在的标签名并确认
- **那么** 系统必须在 `tags` 表插入一条 `archived=false` 的记录，并将其关联到当前快照

#### 场景:同名标签大小写不敏感唯一
- **当** 用户尝试创建 `年终奖`，而表中已存在 `年终奖`（或 `年终奖 ` 带尾空格）
- **那么** 系统必须复用已有标签，而不是新建重复记录

#### 场景:重新启用已归档标签
- **当** 用户新建的标签名在大小写不敏感下匹配一条 `archived=true` 的现有标签
- **那么** 系统必须将该记录 `archived` 重置为 `false`，复用其 id，并关联到当前快照

#### 场景:尝试硬删正在使用的标签
- **当** 用户在标签管理中对一个仍被至少一条快照引用的标签执行"删除"操作
- **那么** 系统必须拒绝硬删，并改为执行归档（软删）

#### 场景:归档标签不出现在候选里
- **当** 用户在新建/编辑快照的标签输入框里查看候选
- **那么** 候选下拉必须仅展示 `archived=false` 的标签
- **并且** 历史快照上已关联的归档标签必须仍然正常显示

---

### 需求:快照与标签多对多关联
系统必须通过关联表 `snapshot_tags` 维护快照与标签的多对多关系，`(snapshotId, tagId)` 作为复合主键。快照硬删时，关联记录必须级联删除（`ON DELETE CASCADE`）。标签在仍被快照引用时不可硬删（`ON DELETE RESTRICT`）。

#### 场景:快照被删除时标签关联清理
- **当** 用户删除一条带有两个标签的快照
- **那么** `snapshot_tags` 中对应 `snapshotId` 的所有行必须被删除
- **并且** `tags` 表本身必须不受影响

#### 场景:批注保存时同步关联
- **当** 用户在批注编辑中把标签从 `[A, B]` 改为 `[A, C]`
- **那么** 系统必须删除 `(snapshotId, B)` 关联，并插入 `(snapshotId, C)` 关联
- **并且** 未改动的 `(snapshotId, A)` 关联必须保持不变

---

### 需求:快照卡片渲染情绪、标签、心得
`SnapshotCard` 必须将情绪 Emoji、已关联的标签集合、非空的心得文本渲染在卡片上，让用户无需展开或编辑即可看见。心得 `note` 为空字符串时必须隐藏该行；已关联标签为空集合时必须隐藏标签行；情绪为 `null` 时必须隐藏情绪图标。

#### 场景:完整批注的快照卡片
- **当** 一条快照的 `mood=4`、`tags=[年终奖, 买房]`、`note="奖金到账..."`
- **那么** 卡片必须显示 🙂、两个标签胶囊、心得文本三行

#### 场景:完全无批注的快照卡片
- **当** 一条快照 `mood=null`、无标签、`note=""`
- **那么** 卡片必须不为批注预留任何可见空白行（不得显示 "-" 占位）

---

### 需求:快照列表按标签筛选
资产快照列表必须支持按标签多选进行筛选。匹配语义为 OR：命中任一选中标签的快照即进入结果集。空选表示不过滤。筛选状态在 ViewModel 中保存，跨冷启动自动重置。

#### 场景:多选标签 OR 过滤
- **当** 用户在筛选条中选中 `#年终奖` 和 `#买房`
- **那么** 列表必须同时包含只含 `#年终奖` 的快照、只含 `#买房` 的快照、以及两者都有的快照
- **并且** 两个标签都不含的快照必须被排除

#### 场景:清空筛选恢复全量
- **当** 用户点击筛选条中的「重置」按钮
- **那么** 选中标签集合必须清空
- **并且** 列表必须显示所有快照，按 `snapshotDate DESC` 排序

#### 场景:冷启动重置筛选
- **当** 用户选中若干标签后关闭应用并重新打开
- **那么** 筛选选中集合必须恢复为空

---

### 需求:标签管理入口
`AssetsScreen` 必须提供一个「标签管理」子入口，允许用户查看所有标签（含已归档），支持重命名和归档/解档操作。被历史快照引用的标签不允许硬删。

#### 场景:重命名标签级联生效
- **当** 用户将标签 `#理财通涨了` 重命名为 `#投资顺利`
- **那么** 所有引用该标签 id 的历史快照卡片必须立即显示新的名称
- **并且** 导出 JSON 时该标签的 `name` 字段必须反映为 `投资顺利`

#### 场景:归档标签
- **当** 用户在标签管理界面对 `#点牛金融` 点击「归档」
- **那么** 该标签 `archived` 必须变为 `true`
- **并且** 历史快照上对该标签的关联不得被破坏

#### 场景:解档标签
- **当** 用户对一条 `archived=true` 的标签点击「解档」
- **那么** `archived` 必须回到 `false`
- **并且** 新建/编辑快照的候选下拉中必须重新出现该标签

---

### 需求:导入导出携带 mood 与标签数据
JSON 导出契约的 `schemaVersion` 必须升级到 `4`，`ExportBundle` 必须新增 `tags: List<ExportTag>` 与 `snapshotTags: List<ExportSnapshotTag>`，`ExportSnapshot` 必须新增 `mood: Int?`。导入时：`schemaVersion >= 4` 的文件必须按完整契约读入；`schemaVersion == 3` 的旧文件必须仍可导入，`mood` 按 `null` 处理、`tags` / `snapshotTags` 按空集合处理，不得报错。

#### 场景:导出新文件包含新字段
- **当** 用户导出 JSON
- **那么** 文件顶层 `schemaVersion` 必须为 `4`
- **并且** 每条快照必须含 `mood`（整数或 null）
- **并且** 文件必须包含 `tags` 与 `snapshotTags` 两个数组

#### 场景:导入旧版本 v3 文件
- **当** 用户导入一个 `schemaVersion=3` 的旧文件
- **那么** 导入必须成功
- **并且** 所有快照的 `mood` 必须为 `null`，新文件的 `tags` / `snapshotTags` 表必须保持为空

#### 场景:导入 v4 文件恢复标签关联
- **当** 用户导入一个 `schemaVersion=4` 文件，其中包含 2 条标签与 3 条 `snapshotTags` 关联
- **那么** 导入后 `tags` 表必须恰好有这 2 条记录
- **并且** `snapshot_tags` 表必须恰好有这 3 条关联
- **并且** 每条快照的 `mood` 必须与文件中的值一致
