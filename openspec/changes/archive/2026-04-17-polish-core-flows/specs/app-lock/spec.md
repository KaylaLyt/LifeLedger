## 新增需求

### 需求:应用密码必须可选

系统必须允许用户在不设置任何本地密码的情况下正常使用 App；当 `AppLockSettingsEntity.passwordHash` 为 `null` 或空字符串时（即 `passwordSet == false`），App 启动后必须跳过 LockScreen，直接呈现主界面。

#### 场景:无密码态启动直接进入
- **当** App 启动且 `passwordSet == false`
- **那么** 系统必须不渲染 LockScreen；主界面必须在首帧直接可交互，无需任何解锁输入

#### 场景:有密码态启动需要解锁
- **当** App 启动且 `passwordSet == true`
- **那么** 系统必须渲染 LockScreen 要求输入密码；只有在 `verifyPassword` 返回 true 时才切换到主界面

### 需求:无密码态必须提供"启用密码"入口

系统必须在 `passwordSet == false` 时，在设置页"数据与安全"分组中显示 `[启用应用密码]` 按钮；点击该按钮必须弹出对话框要求用户输入"新密码"与"确认新密码"，不得要求输入"当前密码"。

#### 场景:设置页按钮随 passwordSet 切换
- **当** 用户位于设置页，且 `passwordSet == false`
- **那么** "数据与安全"分组中必须仅显示 `[启用应用密码]` 按钮，不得显示"修改密码"或"关闭密码"按钮

#### 场景:启用密码对话框不要求当前密码
- **当** 用户点击 `[启用应用密码]` 按钮
- **那么** 系统必须弹出对话框，表单仅包含"新密码"与"确认新密码"两个输入框；不得显示"当前密码"输入框

#### 场景:启用密码校验规则
- **当** 用户在启用密码对话框中提交，且新密码长度 >= 4 且两次输入一致
- **那么** 系统必须将该密码哈希写入 `AppLockSettingsEntity.passwordHash`，同时将 `failedAttempts` 置为 0，`passwordSet` 必须立即变为 `true`

#### 场景:启用密码输入不合法
- **当** 用户在启用密码对话框中提交，但新密码长度 < 4 或两次输入不一致
- **那么** 系统必须拒绝保存，并以 snackbar 文案提示错误原因；`passwordSet` 必须保持 `false`

### 需求:有密码态必须提供"修改密码"与"关闭密码"两种操作

系统必须在 `passwordSet == true` 时，在设置页"数据与安全"分组中同时显示 `[修改密码]` 与 `[关闭密码]` 两个按钮，覆盖有密码态下的全部密码管理需求。

#### 场景:设置页按钮组合
- **当** 用户位于设置页，且 `passwordSet == true`
- **那么** "数据与安全"分组中必须同时显示 `[修改密码]` 与 `[关闭密码]` 两个按钮，且不得显示"启用应用密码"按钮

#### 场景:修改密码需要验证当前密码
- **当** 用户点击 `[修改密码]` 按钮
- **那么** 系统必须弹出对话框，表单包含"当前密码 / 新密码 / 确认新密码"三个输入框；只有在 `verifyPassword(current) == true` 且新密码长度 >= 4 且两次新密码一致时，系统才更新 `passwordHash` 并提示"密码已更新"

#### 场景:关闭密码需要验证当前密码
- **当** 用户点击 `[关闭密码]` 按钮
- **那么** 系统必须弹出对话框仅要求输入"当前密码"；在 `verifyPassword(current) == true` 时，系统必须将 `passwordHash` 清空（设为 `""`）、将 `failedAttempts` 置为 0，`passwordSet` 必须立即变为 `false`

#### 场景:关闭密码不要求二次确认
- **当** 用户在关闭密码对话框中输入正确的当前密码并点击确认
- **那么** 系统必须直接执行关闭操作；不得再追加"你确定要关闭密码吗"这类二次确认弹窗

### 需求:连续 5 次解锁失败必须销毁数据并自动恢复至首次启动态

系统必须在 LockScreen 连续 5 次输入错误密码后触发"数据销毁"流程：清空所有 Room 表、清空 DataStore 偏好、将 `passwordSet` 重置为 `false`；销毁完成后用户必须能在下一次启动（及当前会话）中正常进入 App，不得出现任何无法进入的死锁状态。

#### 场景:错 5 次触发销毁
- **当** 用户在 LockScreen 连续 5 次输入错误密码
- **那么** 系统必须依次执行：1) `database.clearAllTables()`；2) `UserPreferences.clear()`；3) 将 `unlockedState` 设为 `true`；4) 向用户展示一条提示"数据已销毁，App 已重置"

#### 场景:销毁后当前会话自动解锁
- **当** 销毁流程完成
- **那么** 系统必须立即切换到主界面，用户必须不被要求再次输入任何密码；DataStore 中的主题模式、输入单位偏好必须回到默认值

#### 场景:销毁后下次冷启动仍可进入
- **当** 销毁流程完成后用户关闭 App 并重新启动
- **那么** 由于 `passwordSet == false`，系统必须按"应用密码可选"需求直接进入主界面，不得显示 LockScreen

#### 场景:销毁计数在成功解锁后清零
- **当** 用户在 LockScreen 成功输入正确密码
- **那么** 系统必须将 `failedAttempts` 重置为 0，避免跨会话累积失败次数

### 需求:导入 JSON 必须同步密码态

系统必须在用户通过"导入数据"功能加载 JSON 文件时，根据 JSON 中 `lock` 块的 `passwordHash` 字段同步本地密码态，确保导入后的 `passwordSet` 值与 JSON 内容一致。

#### 场景:导入带密码的 JSON
- **当** 用户导入一份 JSON，其 `lock.passwordHash` 字段非空
- **那么** 系统必须把该 `passwordHash` 写入本地 `AppLockSettingsEntity`，`passwordSet` 必须为 `true`；当前会话保持已解锁状态（不强制回退到 LockScreen），下一次冷启动必须按"有密码态"走 LockScreen 流程

#### 场景:导入无密码的 JSON
- **当** 用户导入一份 JSON，其 `lock` 块缺失或 `passwordHash` 为空
- **那么** 系统必须把本地 `passwordHash` 清空，`passwordSet` 必须为 `false`；用户可在设置页看到 `[启用应用密码]` 按钮，可通过该入口重新设置密码
