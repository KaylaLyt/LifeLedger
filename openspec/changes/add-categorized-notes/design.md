## 上下文

当前应用已经具备资产快照、送礼台账、轻量 Todo、本地密码锁和 JSON 导出能力，但没有承接“随时记录想法”的独立模块。用户现阶段主要依赖微信收藏来记录零散内容，问题不在于输入，而在于后续无法以稳定的分类结构回看与整理。

这个新能力仍然运行在同一个离线 Android 应用内，遵循现有的本地优先、SQLite/Room 主存储和单文件 JSON 导出思路。与 Todo 不同，分类笔记不承载行动语义；它的核心行为是纯文本记录、按分类浏览、按更新时间回看以及低成本整理。

## 目标 / 非目标

**目标：**
- 提供一个分类列表首页，展示“未分类”入口和所有真实分类的笔记数量。
- 提供无标题、纯文本、多行正文的笔记录入与编辑能力，并禁止保存空内容。
- 支持笔记归属单个分类，允许在保存笔记时临时创建新分类。
- 支持真实分类默认按新增时间置顶、拖动排序并持久化自定义顺序。
- 支持按分类查看笔记，并以更新时间倒序展示。
- 支持删除真实分类时将其下笔记回退到未分类，以及在单个分类中批量删除笔记。
- 将分类和笔记内容纳入单文件 JSON 导出结构。

**非目标：**
- 不支持标题、富文本、Markdown、附件、图片或语音。
- 不支持多级分类、标签系统、分类重命名或跨分类批量删除。
- 不以搜索、知识图谱或置顶机制作为核心入口。
- 不将笔记自动转为 Todo，也不引入云同步或协作。

## 决策

### 1. 用两个新表建模真实分类与笔记，未分类通过 `categoryId = null` 表达

- 选择原因：未分类是一个虚拟视图而非真实分类，使用可空外键可以避免额外维护一个“系统分类”记录，也天然满足“不可删除”。
- 备选方案：预置一个固定的“未分类”分类记录。
- 不选原因：会引入系统记录与用户分类混用的问题，删除保护、展示名称和导出语义都更复杂。

### 2. 真实分类使用显式 `sortOrder` 持久化顺序，新增分类直接插到顶部

- 选择原因：用户既要“默认按时间倒序，新建在最上面”，又要“支持拖动排序”；显式顺序字段可以稳定表示两种行为。
- 备选方案：只按创建时间排序，或在拖动后临时缓存顺序。
- 不选原因：前者无法持久化拖动结果；后者会在重启后丢失顺序。

### 3. 分类首页固定展示未分类入口，真实分类单独排序

- 选择原因：未分类是笔记回退和临时收纳的默认去处，必须稳定可见；将其固定为虚拟入口，可以避免被拖动排序打乱。
- 备选方案：让未分类参与真实分类排序。
- 不选原因：会使虚拟入口行为与真实分类混淆，且不利于用户形成稳定预期。

### 4. 笔记列表不建标题字段，直接展示正文预览

- 选择原因：用户明确不需要标题，正文预览更符合“想到就记”的低摩擦输入方式。
- 备选方案：为列表额外生成标题摘要字段。
- 不选原因：会引入额外的数据同步逻辑，且与用户的真实需求不一致。

### 5. 分类删除采用“解除归类”而非级联删除

- 选择原因：删除分类并不代表删除内容，笔记应自动回到未分类，保持数据安全和整理弹性。
- 备选方案：删除分类时同步删除其下所有笔记，或阻止删除非空分类。
- 不选原因：前者风险过高；后者增加操作阻力，不符合轻量整理目标。

### 6. 单文件 JSON 导出扩展为包含真实分类与笔记数据

- 选择原因：应用仍需维持“一个 JSON 文件可带走全部核心数据”的承诺，分类与笔记必须进入统一导出结构。
- 备选方案：分类笔记单独导出，或完全不导出该模块。
- 不选原因：会打破当前统一导出体验，也不利于长期迁移。

## 风险 / 权衡

- [分类不能重命名，输错名称后需要新建再迁移] → 录入时必须校验重名并在创建前明确提示，降低误建概率。
- [拖动排序与默认新增置顶并存，顺序规则容易混淆] → 使用显式 `sortOrder` 并保持“未分类固定、真实分类可拖动、新增分类始终置顶”的简单模型。
- [无标题列表在长文本场景下辨识度可能下降] → 列表统一显示正文前段内容和更新时间，确保最低限度可扫描。
- [批量删除存在误删风险] → 将批量删除限制在当前分类内，并要求显式进入选择模式后再确认删除。
- [导出结构新增字段后需要保持长期兼容] → 在导出根对象中新增分类与笔记字段时同步提升 `schemaVersion`，并保留已有字段不变。

## Migration Plan

1. 为数据库新增真实分类表和笔记表，并添加从现有版本迁移到新版本的 Room migration。
2. 在 UI 上新增“分类笔记”入口与分类列表首页，同时保留现有资产、送礼和 Todo 模块。
3. 扩展 Repository 与导出模型，将分类和笔记数据纳入统一读取、写入和导出流程。
4. 为分类唯一性、未分类回退、顺序持久化和批量删除补充测试。
5. 若用户已经在旧版本中导出过 JSON，新版本继续兼容原字段，并通过新的 `schemaVersion` 区分是否包含笔记数据。
